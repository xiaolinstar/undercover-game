# 微信小程序 0.0.3 版本规划

本文定义微信小程序 `0.0.3` 的目标、范围与实现约束，重点解决词库难度运营的可持续性问题，同时保持一期产品“轻量、即开即用、界面负担低”的特点。

## 一、版本定位

`0.0.3` 仍属于一期内的小版本迭代，不引入完整服务端能力，也不改变核心玩法流程。

本版本目标不是做“大改版”，而是在尽量不增加用户视觉负担的前提下，为词库运营补足以下基础能力：

- 难度从离散标签升级为可运营的数值体系
- 用户侧仍只看到简单清晰的三级难度
- 房主反馈可驱动词对难度与质量的持续微调
- 数据结构兼容后续服务端与运营后台

## 二、版本目标

### 1. 产品目标

- 让词库难度分布更细，便于持续扩词与排序
- 让用户反馈真正影响后续词对运营
- 保持当前页面结构稳定，不新增明显学习成本
- 为 `0.1.0` 的服务端词库管理提前收敛数据模型

### 2. 体验目标

- 用户感知上仍然是“简单 / 适中 / 困难”三档
- 不新增复杂筛选面板、复杂图表或重型后台入口
- 首页、词对页、房主页仅做轻量信息优化，不做大改版

## 三、核心思路

### 1. 展示层与运营层分离

难度在系统内部采用数值分数，在用户界面继续使用三级标签。

- 内部字段：`difficultyScore`
  - 取值范围建议为 `1 ~ 100`
  - `1` 代表最容易
  - `100` 代表最难
- 用户展示字段：`difficultyLevel`
  - 仅展示：
    - `easy`
    - `medium`
    - `hard`

这样做有两个好处：

- 用户端足够简单，不会因为分数制增加理解负担
- 运营端具备更高区分度，后续可做排序、推荐、降权、专题分发与自动调档

### 2. 反馈作用于分数，不直接作用于展示文案

用户反馈不应该直接把词对从 `easy` 硬切到 `hard`，而是先调整数值分数，再由分数映射到展示档位。

建议原则：

- “太容易猜”：
  - 说明这组词区分度不够、太直白或当前分数偏低
  - 应适度提高 `difficultyScore`
- “太难描述”：
  - 说明这组词不利于游戏表达或当前分数偏高
  - 应适度降低 `difficultyScore`
- “刚刚好”：
  - 不强制改档
  - 可轻微提升 `qualityScore`
  - 作为后续推荐排序的正向信号

### 3. 页面不大改，只做轻调整

本版本明确限制页面改动规模：

- 不新增复杂运营页面
- 不增加新的主流程页面
- 不在首页新增明显的数据运营模块
- 不把数值难度暴露给普通用户

可接受的调整仅包括：

- 词对列表中继续展示三级难度标签
- 词对排序更稳定、更合理
- 房主反馈文案与反馈结果提示优化
- 词对页增加轻量筛选或排序时，必须保持默认路径不变

## 四、范围定义

## 4.1 本次纳入范围

- 词对数据结构升级：
  - 新增 `difficultyScore`
  - 保留用户端三级难度展示能力
- 词对难度映射规则落地
- 基于反馈的难度分数微调策略
- 基于反馈的质量分与状态运营策略梳理
- 本地存储结构兼容升级
- 为后续远程词库预留同步字段

## 4.2 本次不纳入范围

- 不引入登录、账号体系
- 不引入真实服务端接口
- 不做运营后台页面
- 不做全量用户反馈聚合分析
- 不做大规模 UI 重构
- 不新增明显复杂的筛选与配置流程

## 五、词库数据模型建议

### 1. 当前推荐字段

词对对象建议逐步收敛为以下结构：

```ts
type WordPair = {
  id: string
  civilian: string
  undercover: string
  label?: string
  source?: 'builtin' | 'custom' | 'remote'
  status: 'active' | 'disabled'
  difficulty: 'easy' | 'medium' | 'hard'
  difficultyScore: number
  qualityScore: number
  tags: string[]
  lastUsedAt: number
  useCount: number
  cooldownRounds: number
  flags: string[]
}
```

其中：

- `difficultyScore` 是运营主字段
- `difficulty` 是展示与兼容字段
- `qualityScore` 用于排序、推荐、降权、停用辅助判断

### 2. 分数映射建议

建议先使用静态区间映射，简单且稳定：

- `1 ~ 33`：`easy`
- `34 ~ 66`：`medium`
- `67 ~ 100`：`hard`

后续如需更精细运营，可再升级为可配置映射规则，但 `0.0.3` 不建议把规则做复杂。

### 3. 默认值建议

建议新词默认按主题给一个基础分，而不是只给三级难度：

- 日常生活、食物、基础动物：`20 ~ 35`
- 常见职业、地点、娱乐、校园：`40 ~ 60`
- 抽象概念、历史、商业、科技：`60 ~ 85`

这样后续新增词对时，天然具备更细的区分度。

## 六、反馈驱动的运营规则

### 1. 房主反馈与难度分联动

建议采用“小步调整、逐步收敛”的策略，避免一次反馈导致难度剧烈跳变。

示例策略：

- `too_easy`
  - `difficultyScore +3`
  - 若连续多次出现，可额外触发 `qualityScore -1`
- `too_hard_to_describe`
  - `difficultyScore -3`
  - 若连续多次出现，可额外触发 `qualityScore -1`
- `just_right`
  - `qualityScore +1`

同时增加边界保护：

- `difficultyScore` 最低不小于 `1`
- `difficultyScore` 最高不大于 `100`

### 2. 质量分与状态联动

建议把“难度”与“质量”区分开处理。

- 难度回答的是：这组词对玩家来说偏容易还是偏难
- 质量回答的是：这组词是否适合继续被推荐、曝光或保留

典型场景：

- 一组词很难，但并不一定质量差
- 一组词很简单，也不一定需要下线
- 只有当词对多次被反馈为不适合游戏时，才考虑降权或停用

因此建议：

- `too_easy` / `too_hard_to_describe` 先主要改 `difficultyScore`
- 连续负向反馈达到阈值后，再影响 `qualityScore`
- `qualityScore` 过低时，可自动进入人工复核候选
- `status = disabled` 仍保留为运营最终手动结果，不建议在 `0.0.3` 自动停用

### 3. 本地阶段的反馈策略

由于 `0.0.3` 暂无服务端，反馈只影响当前设备上的词库运营结果。

这意味着：

- 同一用户会感觉“越用越懂我”
- 但全局用户之间不会共享反馈结果

这是本阶段可接受的产品取舍，因为：

- 实现成本低
- 不影响离线可玩
- 可以先验证反馈模型是否合理

## 七、兼容后期服务端的设计

虽然 `0.0.3` 不引入服务端，但数据结构要尽量兼容后续演进。

### 1. 字段兼容原则

本地与未来远端尽量共用同一套核心字段：

- `id`
- `status`
- `difficultyScore`
- `difficulty`
- `qualityScore`
- `tags`
- `flags`
- `updatedAt`（后续可增加）
- `version`（后续可增加）

### 2. 同步策略预留

未来接服务端时，建议采用：

- 内置词库作为兜底
- 远端词库作为覆盖层
- 本地反馈作为临时增量层

可演进为三层合并模型：

1. 内置默认词库
2. 远端下发的运营词库元数据
3. 用户本地尚未上报或尚未同步的反馈结果

这样有几个优势：

- 服务端未返回时仍能正常游戏
- 远端可以统一控制词对上下线
- 本地可以保留即时反馈体验

### 3. 后台管理能力预留

后续服务端后台建议至少支持以下能力：

- 新增词对
- 编辑词对
- 调整 `difficultyScore`
- 调整 `qualityScore`
- 上下线词对
- 配置标签、主题、专题词包
- 查看负向反馈趋势

但这些不应提前出现在 `0.0.3` 的用户界面里。

## 八、页面与交互约束

### 1. 首页

- 继续保持轻量开局工具定位
- 不增加运营面板式信息
- 词对预览仍以少量精选为主

### 2. 词对页

- 保留当前列表结构
- 难度继续显示三级标签，不显示数值分
- 如增加排序，优先使用默认智能排序，不要求用户手动理解分数规则

### 3. 房主页

- 保留现有三种反馈按钮
- 可优化反馈成功后的说明文案
- 不展示难度分变化过程

整体原则：

- 用户只需要知道“这组词简单 / 适中 / 困难”
- 不需要理解你们后台是怎么运营的

## 九、建议验收标准

- 词对数据已支持 `difficultyScore`
- 用户侧仍只显示三级难度，不暴露数值分
- 反馈提交后可正确影响本地词对难度分或质量分
- `difficultyScore` 改变后，`difficulty` 能自动重新映射
- 默认词库能按新的难度策略初始化
- 页面结构无明显新增复杂模块
- 首页、词对页、房主页视觉负担不高于 `0.0.2`

## 十、版本产出建议

`0.0.3` 建议拆成三个实施小项，降低改动风险：

### A. 数据模型升级

- 增加 `difficultyScore`
- 增加难度映射工具函数
- 调整默认词库初始化规则

### B. 反馈策略升级

- 将反馈逻辑改为优先作用于 `difficultyScore`
- 对 `qualityScore` 只做轻量辅助调整

### C. 页面轻量适配

- 保持原有页面结构
- 只调整文案、标签与排序逻辑

## 十一、版本结论

`0.0.3` 的最佳方向，不是提前做出完整后台，而是先把“词库难度运营模型”打磨清楚。

优先把下面四件事做扎实：

- 内部难度分数化
- 外部展示三级化
- 反馈驱动微调
- 数据模型兼容服务端

这样到 `0.1.0` 再接入服务端与运营后台时，前后端都会更稳定，用户端也不需要再经历一次大的交互变动。
